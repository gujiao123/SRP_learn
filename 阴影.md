### 1. 为什么偏偏是缩“半个”像素（0.5），而不是 1 个？

这跟 GPU 硬件底层的**“双线性过滤（Bilinear Filtering）”权重算法**有直接关系。

在 GPU 的眼里，像素不是一个小方块，而是**一个个带中心点的网格**：
* 像素的边缘坐标是整点的边缘（比如 `0.0`，`1.0`）。
* 像素的**正中心**坐标是在 `0.5` 的位置（比如 `0.5`，`1.5`）。

当 GPU 进行平滑采样（双线性过滤 / 硬件 PCF）时，它会在采样点周围画一个 2x2 的框，然后根据距离分配权重：
* 如果你采样的 UV 刚好落在**两个像素交界的边缘（距离边缘 0.0 个像素）**：GPU 会认为你两边都想要，于是分配 **50% 像素 A + 50% 像素 B**。这就是产生渗色（Bleeding）的元凶！
* 如果你采样的 UV 刚好落在**某一个像素的正中心（距离边缘 0.5 个像素）**：GPU 会认为你完全就只想要当前这个像素，于是分配 **100% 像素 A + 0% 像素 B**。

所以，我们往里收缩 `0.5` 个像素的真实含义是：**把采样的极限边界，死死地钉在最外沿那一圈像素的正中心上！**
只要采样游标不超过像素的中心点，硬件级别的混合算法就绝对不会去伸手摸隔壁邻居的数据。如果缩 1 个像素，那就白白浪费了半个像素的阴影清晰度，0.5 是在不越界前提下的极限“榨干”利用率。

### 2. 对于 PCF 多次采样，周边的安全区外还能采到吗？

这个问题极其敏锐！如果我开了 3x3 或者 5x5 的 PCF，我的采样网格是很大的，岂不是就算中心点被钉在安全区，边缘的采样点还是会越界伸到隔壁 Tile 去？

这就涉及到了配合 C# 的 **Shader 端（HLSL 代码）中的魔法限制**。

这个 `data` 算出来的安全框（`[0.250488, 0.499511]`），会作为参数传给 Shader。
在 Shader 真正进行 PCF 采样（比如循环 9 次或者 25 次）的时候，每一次偏移后的 UV，**都会被强行 Clamp（截断）限制在这个安全区里**：

```hlsl
// Shader 里的伪代码大概是这样：
float2 pcfUV = baseUV + offset[i]; // PCF 需要向外偏移去采周边
pcfUV = clamp(pcfUV, data.xy, data.xy + data.z); // 撞墙！不准超出安全区！
```

**所以会发生什么奇妙的现象？**
* **安全区“内”的像素**：正常被 PCF 采到，产生完美的柔和过渡。
* **企图越界跨过安全区（哪怕只过 0.1 像素）的 PCF 采样**：会被 `clamp` 函数像一堵空气墙一样死死挡住。挡住后，它只能重复去采安全区**“最边缘”那个像素**的值。
* **结果**：对于那些靠近边界的阴影，原本应该很柔和地向外扩的 PCF，在屏幕边界处会稍微变硬一点（因为它重复采样了边缘像素，没有外面的数据给它糊了），但是**绝对不会采到隔壁别人家（比如点光源 B）的异常数据而产生黑线斑块**。

这就是图形学里非常经典的**利用 Clamp To Edge 逻辑结合 Shader 人为制造“隔离带”**的做法！安全区外的物理像素其实就是隔壁灯的像素了，所以我们用空气墙（Clamp）禁止一切触碰。

但在 GPU 底层的内存和数学模型里，像素（Texel）**根本不是一块面积，而是一个个没有大小的“数据点（Data Point）”**！

我们要完成一次彻底的**思维转换**。请看着下面这幅图我一步步讲：

### 1. GPU 眼里的“像素”长什么样？

假设有一张宽 3 个像素的贴图（比如像素 A、B、C）。
在我们的脑海里，它是三个并排的方块。但在 GPU 的坐标系（UV 或者整数坐标）里，它是这样定义的：

```text
坐标轴: 0.0          1.0          2.0          3.0  (这是 float 浮点数刻度)
        |            |            |            |
        |            |            |            |
脑海里的: ▏  像素 A    ▏  像素 B    ▏  像素 C    ▏ (一块块的面积)
实际数据: ▏     ●      ▏     ●      ▏     ●      ▏ (真正存颜色的数据点！)
        |    0.5     |    1.5     |    2.5     |
        |            |            |            |
```

你看懂了吗？
* **整数（0.0, 1.0, 2.0）**：其实是方块与方块之间的**物理边界（缝隙）**。这里其实是**没有数据**的！
* **含有 .5 的坐标（0.5, 1.5, 2.5）**：才是每一块像素的**正中心**！在这里才真正存放着你读取的颜色或深度数据。

这就是为什么 UV 采样用的是 `float`，因为它是一把连续的尺子。

### 2. 双线性过滤（Bilinear）是如何计算的？

当我们给 GPU 传递一个 `float` 坐标去采样时，GPU 其实是在做一道**距离混合题**。

**情况一：你要求采样整数边界 `1.0`**
你想着：“1 这个整数肯定是完美对应第一个像素的边界对吧？”
但 GPU 拿到坐标 `1.0`，它找了找附近真理存在的地方，发现：
> “哎呀，`1.0` 的左边 0.5 距离处有一个数据点（像素 A 的中心 0.5），右边 0.5 距离处也有一个数据点（像素 B 的中心 1.5）。距离一样近！”
**GPU 的判定**：既然距离一样近，那我就把 A 取 50%，B 取 50%，揉在一起给你！
👉 **结果**：这就是著名的边缘渗色（Bleeding），画面在这条缝隙上变糊了，混入了隔壁的东西。

**情况二：你要求采样像素正中心 `0.5`**
你给 GPU 传坐标 `0.5`。
GPU 找了找附近的数据点：
> “哇！你踩的坐标 `0.5` 脚下，刚好就结结实实踩着像素 A 的数据点！至于像素 B 的数据点，在 `1.5`，离得有 1.0 那么远呢！”
**GPU 的判定**：你绝对是只想要 A。好，我给你 100% 的 A，0% 的 B！
👉 **结果**：得到了纯净无比、不掺任何杂质的单个像素数据。

现在，假设你的阴影图集（Atlas）被切开了：左边属于灯 1（安全区），右边属于灯 2（禁区）。
它们的**地界线**，正好划在了整数坐标 **`1.0`（缝隙处）**。

如果你计算出来的 UV 坐标是 `1.0`（也就是原原本本地紧贴地界线），GPU 就去采样了。
结果就像刚才的情况一：GPU 会给你 **50% 的灯1边缘 + 50% 的灯2边缘**。
你的灯 1 的阴影边缘，硬生生混入了隔壁灯 2 测试出来的阴影数据，画面上立马出现一条黑线瑕疵！

**怎么办？**
我们必须下死命令告诉 Shader：**“你采样的探针，最死角最极限只能移动到属于我的那个像素的正中心点！不准再往边界靠近了！”**

既然边界线是 `1.0`，我这个最边缘像素的中心点在哪里？
在 `1.0 - 0.5 = 0.5` 的地方！

*   你要 **0.5**（点对点）：我给你只返回一个值，**100% 的纯红**。
