## 一、自阴影（Shadow Acne）与法线偏移

### 1.1 自阴影是怎么产生的？

阴影贴图本质是一张**离散的深度图**——分辨率有限，所以存的深度值不是连续的曲面，而是**一层层台阶**。

```text
真实斜面：          阴影贴图记录的深度（台阶状）：

    ╱                    ┌─┐
   ╱                  ┌─┘ │
  ╱                ┌─┘   │
 ╱              ┌─┘     │
╱            ───┘       │
```

当我们拿片元的实际深度和阴影贴图的台阶深度做比较时：
- 台阶**凸出来的部分** → 片元在台阶后面 → 被判定为"在阴影里" ❌
- 台阶**凹进去的部分** → 片元在台阶前面 → 被判定为"不在阴影里" ✅

结果就是斜面上出现了**条纹状的明暗交替**——这就是自阴影（Shadow Acne）。

### 1.2 法线偏移怎么解决的？

把片元沿法线方向**往外推一点点**，推出台阶的"凸包"范围，这样比较深度时就不会被自己的台阶挡住了。

```text
原始位置 A：被台阶包裹 → 误判为阴影
         ↓ 沿法线外推
新位置 A'：跳出台阶 → 正确判定为无阴影

推多远？最远需要覆盖到台阶的对角线 → √2 × texelSize
```

### 1.3 透视投影下，法线偏移为什么要随距离缩放？

方向光是正交投影，台阶在所有距离上**大小一样**，所以法线偏移是固定值。

但聚光灯/点光源是**透视投影**，离光源越远，一个像素覆盖的世界空间越大 → 台阶越大 → 需要更大的法线偏移：

```text
光源 ◆━━━━━━━━▶

距离1：|█|█|█|  台阶小，偏移小
距离2：|░░|░░|  台阶大2倍，偏移也大2倍
距离3：|▓▓▓|▓▓▓| 台阶大3倍，偏移也大3倍
```

**C# 端**算出"距离1处"的偏移量（存在 `tileData.w`），**Shader 端**实时乘以实际距离：

```hlsl
float distanceToLightPlane = dot(surfaceToLight, lightPlane);
float3 normalBias = normal * (distanceToLightPlane * tileData.w);
//                            ↑ 实际距离              ↑ 距离1处的偏移
```

## 二、GPU 纹理采样的底层真相

### 2.1 像素不是方块，是数据点

在我们脑海里，像素是一块块色块。但**GPU 内部的数学模型里，像素是没有面积的数据点**，钉在每个格子的正中心：

```text
坐标轴: 0.0          1.0          2.0          3.0
        |            |            |            |
脑海里: ▏  像素 A    ▏  像素 B    ▏  像素 C    ▏  ← 色块（面积）
GPU里:  ▏     ●      ▏     ●      ▏     ●      ▏  ← 数据点（无面积）
        |    0.5     |    1.5     |    2.5     |
```

- **整数坐标（0.0, 1.0, 2.0）**= 像素之间的**缝隙**，这里没有原始数据
- **x.5 坐标（0.5, 1.5, 2.5）**= 像素的**正中心**，真正存数据的地方

### 2.2 双线性过滤 = 距离混合

GPU 拿到一个 float UV 后，找周围最近的数据点，按距离混合：

```text
数据点：    ● 纯红(0.5)              ● 纯蓝(1.5)
            |                        |
采样 0.5  → 100% 红 + 0% 蓝   = 纯红        ← 踩中数据点
采样 0.8  → 70% 红 + 30% 蓝   = 偏红紫      ← 偏左
采样 1.0  → 50% 红 + 50% 蓝   = 纯紫        ← 缝隙上，两边各半
采样 1.2  → 30% 红 + 70% 蓝   = 偏蓝紫      ← 偏右
采样 1.5  → 0% 红 + 100% 蓝   = 纯蓝        ← 踩中数据点
```

> **关键认知**：UV 采样是连续的 float，不是离散的 int。GPU 在任意位置都能通过距离插值"造出"一个值。

## 三、Tile 边界控制（防止采样越界）

### 3.1 为什么需要边界控制？

点/聚光灯的阴影都画在同一张 Atlas 大图上，多个 tile 紧挨着：

```text
Atlas（UV空间 [0,1]×[0,1]）：
┌────────────┬────────────┐
│   tile 0   │   tile 1   │  ← 紧挨着，没有间隔！
│  (聚光灯A) │ (点光+X面) │
├────────────┼────────────┤
│   tile 2   │   tile 3   │
└────────────┴────────────┘
```

PCF 滤波在目标像素周围多采几个点，如果片元在 tile 最边缘，采样点会**跨到隔壁 tile**：

```text
     tile 0         │     tile 1
                    │
              · · · │ · ·
              · · ☆ │ · ·   ← PCF 采样点越界了！
              · · · │ · ·     读到了另一盏灯的深度 → 假阴影！
```

方向光靠**级联球体**天然限制了采样范围，不需要人为处理。但点/聚光灯没有这个保护。

### 3.2 解决方案：安全区 + Clamp

**C# 端**为每个 tile 算一个"安全区"（四边各内缩半个像素）：

```csharp
// SetOtherTileData()
float border = atlasSizes.w * 0.5f;  // 半个像素的 UV 大小
data.x = offset.x * scale + border;  // 安全区左边（向右收缩）
data.y = offset.y * scale + border;  // 安全区下边（向上收缩）
data.z = scale - border - border;    // 安全区宽度（两边各减半像素）
```

**具体数字**（Atlas=1024, split=4, tile 5, offset=(1,1)）：

```text
完整tile范围: [0.25, 0.50] × [0.25, 0.50]    宽度 = 0.25
安全区范围:   [0.250488, 0.499511] × 同上     宽度 = 0.249023

 0.50 ─────────────────┐ 完整 tile 边框
      │                │
      │ ╔════════════╗ │ ← 0.499511（右边往左缩半像素）
      │ ║  安全区    ║ │
      │ ║            ║ │
      │ ╚════════════╝ │ ← 0.250488（左边往右缩半像素）
      │                │
 0.25 └────────────────┘
```

**Shader 端**每次采样前强制 Clamp 到安全区内：

```hlsl
float SampleOtherShadowAtlas(float3 positionSTS, float3 bounds) {
    // bounds.xy = 安全区左下角, bounds.xy + bounds.z = 安全区右上角
    positionSTS.xy = clamp(positionSTS.xy, bounds.xy, bounds.xy + bounds.z);
    return SAMPLE_TEXTURE2D_SHADOW(_OtherShadowAtlas, SHADOW_SAMPLER, positionSTS);
}
```

PCF 的**每个采样点单独 Clamp**，不是只 clamp 中心点。越界的采样点会被"钉"在安全区边缘，重复采同一个像素（阴影稍微变硬），但**绝不会采到隔壁 tile 的数据**。

### 3.3 为什么恰好缩半个像素？

这是**最极限的安全距离**：

```text
tile 边界在 1.0（整数位置 = 像素缝隙）
最边缘像素的中心在 0.5

如果允许采到 1.0 → GPU 混合 50%自己 + 50%隔壁 → 渗色 ❌
如果 clamp 到 0.5 → GPU 给你 100%自己 + 0%隔壁 → 纯净 ✅
如果 clamp 到 -0.5（缩1整个像素）→ 最外圈像素永远采不到 → 浪费分辨率 ❌

0.5 = 保证不越界的前提下，最大化利用每一个像素
```

## 四、总结对比

| | 方向光 | 聚光灯/点光源 |
|---|---|---|
| **投影类型** | 正交 | 透视 |
| **法线偏移** | 固定值 | 随距离线性缩放 |
| **采样时除 w** | 不需要（w=1） | 必须（透视除法） |
| **边界保护** | 级联球体（天然） | 手动 Clamp（SetOtherTileData） |
| **Shadow Pancaking** | 开启 | 关闭（否则灯后面物体被压扁） |
| **级联** | 有（2~4级） | 无（每灯就一层） |
| **点光源** | — | 6个面（cubemap），FOV固定90° |

## 五、点光源近处阴影消失（近平面裁剪）

### 5.1 现象

球体靠近点光源时，投下的阴影会逐渐消失。从 Frame Debugger 看阴影贴图，球体只呈现出**月牙形**，而不是完整圆形。

### 5.2 原因：相机近平面把球切掉了

**重要认知：点光源的"球形"只是照明模型，渲染阴影时实际上是放了 6 台透视相机！**

```text
照明模型（球形）：          阴影渲染（6台相机）：
                               相机+Y
      ↖ ↑ ↗                      ↑
      ← ◆ →  光照=球形         ← ◆ →  每个方向一台透视相机
      ↙ ↓ ↘                      ↓
                               相机-Y ...共6个
```

每台相机都有**近平面（默认0.1）**和**远平面（= Light.Range）**：

```text
从+Y方向的相机看：

    光源 ◆
         ↓ 
    [近平面 0.1]
    ────────────    ← 球的上半部分（靠近光源那侧）在近平面内！
        )           ← 只有这个月牙（下半面）被渲染进深度图
    ────────────
        地板
```

- 球的**靠近光源那一半** → 在近平面之内 → 被 GPU 剔除 → 深度图里没有它
- 地板采样阴影时 → 对应位置没有遮挡物 → **判定为不在阴影里 → 亮的**
- 球离光越近 → 月牙越细 → 遮挡越少 → 阴影消失越明显

### 5.3 为什么不把近平面调成 0.0001 来修复？

这涉及 **Z-buffer 精度**的根本矛盾。GPU 的深度缓冲是双曲线分布，不是线性分布：

```text
近平面=0.1，far=20 时：         近平面=0.0001，far=20 时：

精度分布良好（靠近有精度）      0.0001~1米之间精度极差！

z(0.5m) = 0.990                 z(0.5m) = 0.9999975
z(1.0m) = 0.980     ← 好分辨   z(1.0m) = 0.9999950  ← 几乎一样！
z(2.0m) = 0.960                 z(2.0m) = 0.9999900  ← Z-Fighting！
```

近平面越小 → near/far 比值越大 → 中远距离的深度值全挤到 [0.999..., 1.0] 这个极小区间内 → **Z-Fighting（深度闪烁、假阴影条纹）**。

### 5.4 结论

| | 近平面大（默认0.1） | 近平面极小（0.0001） |
|---|---|---|
| **近处物体** | ❌ 被裁剪，阴影消失 | ✅ 可见 |
| **远处精度** | ✅ 充足 | ❌ Z-Fighting |

这是**物理上无法完全解决的权衡**。点光源近处阴影消失是已知限制，Unity 和教程都没有提供完美方案。实际项目中通常通过避免物体贴近光源、或关闭点光源实时阴影改用烘焙来规避。
