## ❓ 我的困惑 & 解答

### Q1：GI 我们自己不用写吗？
**答**：分两件事。烘焙算光（确定光线怎么在空间里弹射）不需要我们写，Unity 底层的 Progressive Lightmapper 帮我们算，结果存为一张贴图（Lightmap）。
**但是**，从贴图里把数据读出来、涂到模型上这件事，必须我们自己写在 Shader (`GI.hlsl`) 里！我们写的是"取光器"，不是"烤光器"。

---

### Q2：为什么 Baked 之后场景全黑了？
**分析链路**：
1. 把 Directional Light 改成 `Baked` 模式 → 引擎底层把这束光的实时电源"拔掉"了（不再传进 `GetDirectionalLightCount()` 的列表里）。
2. 我们的 `Lit.shader` 只有实时光的 for 循环代码 → 循环次数变成 0 → 结果 = `颜色 * 0 = 纯黑`。
3. 虽然烤好的光照贴图躺在硬盘上 → 但 Shader 里根本没写去读取它的代码 → 视觉上等于瞎子。

**结论**：实时光没了，烘焙光的读取代码又还没写，所以纯黑。这是代码不完整的正常状态，不是 Bug。

---

### Q3：为什么要勾选 Static？
**核心原因**：光照烘焙的本质就是把光照和阴影"像文身一样印在物体表面"（存成 Lightmap 贴图）。
- 如果物体能动 → 以后把箱子踢飞了，地上的大黑影还印在原地 → 极其出戏的穿帮！
- 所以 Unity 的烘焙器**拒绝处理**没有打上 `Static (Contribute GI)` 标记的可动物体。
- 打上 `Static` = 你在和引擎签署「我发誓这辈子绝对不移动」的契约。

**问：动态角色怎么办？** → 靠 Light Probes (光照探针)。

---

### Q4：三种灯光模式是什么意思？
| 模式 | 比喻 | 特点 |
|---|---|---|
| **Realtime** | 手电筒（每帧实时算） | 高自由度，性能消耗大，无真实反弹光 |
| **Baked** | 老照片（预先算好印死） | 性能开销为零，但光是死的，动态物体没有阴影 |
| **Mixed** | 万金油：直射光实时，反弹光烘焙 | 业界最常用，兼顾品质与性能 |

**关键理解**：
- `Realtime` 的光会被发进 `visibleLights` 数组 → 我们的 `SetupLights()` 能拿到 → for 循环能算。
- `Baked` 的光被引擎底层从这个数组里**踢出去**了 → 必须走光照贴图/探针另一条路读取。

---

### Q5：为什么非静态的球没有颜色变化，只是变亮了一点？
**根本症结**：烘焙大师（Progressive Lightmapper）根本不认识我们自己取名的 `_BaseColor`！

世界上有**两个互不通气的独立大脑**同时在工作：
- **大脑 1（显卡画师）** = 我们写的 `LitPass.hlsl` → 用 `_BaseColor` 把墙涂成漂亮的蓝色显示在屏幕上 ✅
- **大脑 2（烘焙大师）** = Progressive Lightmapper → 去材质找默认接头暗号 `_Color`，找不到 → 全部当白色算反弹光 ❌

结果：屏幕上墙壁看起来是蓝色（大脑1的成果），但光照贴图和空气探针里存的全是"白墙反弹的白光"（大脑2算的结果）→ 动态小球吸收到的全是白光 → 不受任何颜色影响！

**解决方案**：写 `Meta Pass` 通道（即将在第 4 节实现），充当两个大脑之间的"翻译外交官"，把真实的 `_BaseColor` 在烘焙期间递给大师。

---

### Q6：为什么地板没有阴影？
**原因**：因为太阳调成了 `Baked` 模式后，实时投影能力被整个收走了。地板如果没勾选 `Static`，烘焙大师也不会去给它算印死在面上的阴影贴图 → 两边都没人管 → 无阴影。

**解决**：地板同样要勾选 `Static` 并重新烤一次。

## 📐 数据传输链路图谱 (光照贴图 UV 的旅程)

```
[引擎底层 C++]
     |
     | CameraRenderer.cs 里声明:
     | perObjectData = PerObjectData.Lightmaps | PerObjectData.LightProbe
     | (告诉引擎：拿光照贴图UV和探针数据一起打包发上高速公路)
     |
     ↓
[Shader 顶点 Attributes 结构体]
     | GI_ATTRIBUTE_DATA
     | → 展开为：float2 lightMapUV : TEXCOORD1;
     | (从顶点的第二套纹理坐标槽里拿到光照贴图的门牌号UV)
     |
     ↓
[顶点着色器 LitPassVertex]
     | TRANSFER_GI_DATA(input, output)
     | → 展开为：output.lightMapUV = input.lightMapUV * unity_LightmapST.xy + unity_LightmapST.zw;
     | (插值传递给片元，并应用物体在光照大图集上的偏移量校正)
     |
     ↓
[片元着色器 LitPassFragment]
     | GI gi = GetGI(GI_FRAGMENT_DATA(input), surface);
     |   GI_FRAGMENT_DATA(input) → 展开为：input.lightMapUV
     | (把坐标+表面法线数据交给专职取光函数)
     |
     ↓
[GI.hlsl 的 GetGI 函数]
     | gi.diffuse = SampleLightMap(lightMapUV)  ← 静态物体读贴图
     |           + SampleLightProbe(surfaceWS); ← 动态物体读探针
     |
     ↓
[Lighting.hlsl 的 GetLighting 函数]
     | float3 color = gi.diffuse * brdf.diffuse; ← 烘焙光 * 材质漫反射
     | color += 实时光 for 循环结果;
     | return color;
```

## 🔥 深度难点：lightMapUV、ST变换、Meta Pass 与 GI.hlsl 的镜像关系

> 这是本章最独特、最容易混淆的概念点！

### 核心事实：光照贴图是一张"大拼图"

Unity 不会给每个物体单独生成一张 Lightmap，而是把场景里所有 Static 物体的光照信息，统一打包塞进一张（或几张）大图集里：

```
整张 Lightmap 大图集 (例如 1024x1024)
┌──────────────────────────────────┐
│   地板的光照区域（左下角）        │
│   ┌──────────┐                   │
│   │ ░▓▓▓░░░ │                   │
│   └──────────┘  屋顶的区域       │
│              ┌───────┐           │
│              │▒▒░░▒▒│           │
│              └───────┘           │
│   左墙的                         │
│   区域（右上角）...              │
└──────────────────────────────────┘
```

每个物体都有唯一的 `unity_LightmapST`（缩放+偏移变量），记录它在这张大图的哪个角落、占多大一块。

---

### Q7：`TRANSFER_GI_DATA` 里的 ST 变换，和 `MetaPassVertex` 里的 ST 变换，有什么关系？

**两段代码：**
```hlsl
// GI.hlsl 里（读取端）
output.lightMapUV = input.lightMapUV * unity_LightmapST.xy + unity_LightmapST.zw;

// MetaPassVertex 里（写入端）
input.positionOS.xy = input.lightMapUV * unity_LightmapST.xy + unity_LightmapST.zw;
```

**回答：数学公式完全一样！但方向完全相反——一个是"存档"，一个是"读档"！**

#### 理解 `input.lightMapUV` 是什么

- 它是模型网格数据里自带的 **第二套 UV（TEXCOORD1）**
- 范围是 0~1，描述模型自身各顶点的"展开相对位置"
- 这就是一张模型在 UV 展开图中的"自我坐标"，不含任何图集信息

#### 为什么两边都需要做 ST 变换

**因为这个 `lightMapUV` 只知道自己的"身份编号"，不知道自己被分配到了图集的哪个角落！**

必须用 `unity_LightmapST` 做一次转换，才能把自身坐标映射到整张图集的真实地址：
```
自身UV [0.3, 0.7]  ×  ST缩放 [0.25, 0.25]  +  ST偏移 [0.5, 0.0]
        ↓
图集地址 = [0.575, 0.175]  ← 真正的像素坐标
```

---

### 🪞 Meta Pass 与 GI.hlsl 的完整镜像图

```
模型顶点数据
    input.lightMapUV = [0.3, 0.7]  ← 模型自带的原始 UV，范围 0~1
              |
              | × unity_LightmapST  （两边用的是完全相同的变换！）
              |
              ↓
       图集地址 = [0.575, 0.175]
              |                         |
    ┌─────────┘                         └────────────┐
    ↓ 烘焙期（点击 Generate）               ↓ 运行时（每帧渲染）
  MetaPassVertex                          GI.hlsl 的 TRANSFER_GI_DATA
  "把这个地址当做裁剪空间位置"          "把这个地址传给片元去采样"
              ↓                                       ↓
  MetaPassFragment                          SampleLightMap(lightMapUV)
  "在这个图集地址：写入红色"            "从这个图集地址：读出红色"
              ↓                                       ↓
        Lightmap 图片                       gi.diffuse = 红色光照
       （存进硬盘）                          （涂到模型上）
```

**存档坐标 = 读档坐标 → 颜色正确 → 色溢正确！**

---

### 📝 一句话总结

> `input.lightMapUV` 两边是同一份原始数据，`unity_LightmapST` 两边是同一个变换。  
> Meta Pass 用它来确定"把颜色写进图集的哪个位置"，  
> GI.hlsl 用它来确定"从图集的哪个位置把颜色读出来"。  
> 这是完美的镜像关系：**生产者写的地址，消费者从同一个地址读取。**

## 🌟 第五节：自发光 (Emissive Surfaces)

### Q8：自发光到底是什么？和普通颜色有什么区别？

普通颜色需要**接受外界的光**才能被看见（没有光源就是黑色）。  
**自发光**是表面自己产生光，不依赖任何光源，比如霓虹灯、岩浆、发光屏幕。

**实现原理极其简单**：在计算完所有光照之后，直接把发光颜色**加**在最终颜色上：
```hlsl
// LitPassFragment 末尾
color += GetEmission(input.baseUV);  // 不乘任何光照系数，朴素的加法！
```

---

### Q9：静态物体也有自发光效果，这正常吗？

**完全正常！** 自发光有两层互相独立的效果：

| 效果层 | 实现位置 | 影响谁 | 静态/动态都有？ |
|---|---|---|---|
| **自己发亮**（屏幕显示） | `LitPassFragment` 加法 | 只影响自身外观 | ✅ 两种都有 |
| **照亮周围**（贡献烘焙） | Meta Pass Y 标志位 | 周围的静态物体 | 只有 Static 物体 |

---

### Q10：`Global Illumination` 下拉框只管自发光，不管烘焙透明度！

> 教程原文："*Despite its name it **only affects baked emission**.*"

- `None` → 自发光不参与烘焙
- `Baked` → 自发光会烤进 Lightmap，照亮周围静态物体
- **和透明度完全无关！**

**烘焙透明度是自动处理的**，烘焙大师根据材质的 Render Queue 判断透明度，不需要手动开关。

## 🪟 第六节：烘焙透明度 (Baked Transparency)

### Q11：为什么半透明物体的烘焙阴影是实心黑色而不是半透明的？

**根本原因**：烘焙大师（Lightmapper）只认识三个硬编码变量名：
```
_MainTex   ← 贴图
_Color     ← 颜色
_Cutoff    ← 裁剪阈值
```

我们的 Shader 用的是 `_BaseMap` 和 `_BaseColor`，烘焙大师不认识 → 当成白色实心方块处理 → 地板上出现完全黑色的死影！

**解决方案**：加两个隐藏马甲属性 `_MainTex` 和 `_Color`，每次材质改动时通过 `CopyLightMappingProperties()` 自动同步真实值给马甲 → 烘焙大师就能正确读到透明度了。

**验证方法**：用带圆形镂空的 Default-Particle 贴图做 Clip 材质，烘焙后地板阴影应该是**圆形**而不是**矩形**！

## 🎱 番外：MeshBall 的光照探针问题

### Q12：为什么 DrawMeshInstanced 的实例接收不到光照探针？

**普通 GameObject**：有 MeshRenderer 组件 → Unity 自动根据 Transform 坐标查探针 → 全自动！

**DrawMeshInstanced**：直接提交给 GPU，没有 GameObject，没有 MeshRenderer → Unity 的自动探针系统视而不见 → 所有实例接收不到任何 GI！

**解决方案 A：手动查探针**
```csharp
// 1. 从变换矩阵第4列提取每个实例的世界坐标
var positions = new Vector3[1023];
for (int i = 0; i < matrices.Length; i++) {
    positions[i] = matrices[i].GetColumn(3); // 矩阵第4列 = 世界坐标
}

// 2. 让 Unity 批量计算 1023 个位置的球谐系数
var lightProbes = new SphericalHarmonicsL2[1023];
LightProbes.CalculateInterpolatedLightAndOcclusionProbes(positions, lightProbes, null);

// 3. 把探针数据填进 Block，随 DrawCall 发给 GPU
block.CopySHCoefficientArraysFrom(lightProbes);
```

**解决方案 B：LPPV（Light Probe Proxy Volume，光照探针代理体积）**
- 在场景里放一个 3D 盒子组件（LPPV），框住所有实例
- Unity 在盒子内部预计算好各点的探针插值数据
- 实例在盒子内移动时，直接查盒子数据，不需要每帧重新计算
- 适合实例**频繁移动**但**位置集中**的场景（粒子系统等）

### 两种方案对比

| | 方案A（手动查探针） | 方案B（LPPV） |
|---|---|---|
| **适合场景** | 实例位置固定，生成后不移动 | 实例频繁移动，但集中在固定区域 |
| **性能** | 实例动了就要重算 | 实例动了直接查预计算数据 |
| **设置** | 纯代码，无需额外组件 | 需要在场景里放 LPPV 组件 |

> MeshBall 的球生成后不移动 → **方案A更合适**，只需计算一次就可以永久复用。



# 📖 第五章完整回顾：烘焙光照 (Baked Light)

## 🤔 为什么需要烘焙光照？

**实时光照的核心痛点**：

每帧 GPU 都要计算光线如何从光源出发、打到物体、再弹射到眼睛。尤其是**间接光**（光线在物体间相互弹射的那部分），计算量极其庞大，实时做根本跑不起来。

**烘焙的思路**：
> "这些墙壁和地板反正一辈子不移动，那我们提前把光照算好，结果存成一张图片（Lightmap），游戏运行时直接读图，快得飞起！"

---

## 🧩 这一章解决了哪些具体问题？

### 问题 1：烤完之后场景全黑了！

**为什么**：把灯光设成 [Baked](cci:1://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Editor/CustomShaderGUI.cs:169:4-189:5) 后，引擎把这盏灯从实时列表里移除了，`GetDirectionalLightCount()` 返回 0，for 循环乘以 0 = 全黑。  
**怎么解决**：在 `GI.hlsl` 里写读取 Lightmap 的代码，在 Shader 里开 `LIGHTMAP_ON` 多变体。  
**效果**：Static 物体的间接光照从贴图里正确读出来。

---

### 问题 2：非静态物体（动态角色）也全黑了！

**为什么**：动态物体没有 UV，不能参与 Lightmap。  
**怎么解决**：场景里放**光照探针**（Light Probes），动态物体根据自身位置，在附近探针之间插值，拿到该位置的 GI 近似值。  
**效果**：动态小球靠近被红光照亮的区域，也会被微微染上红色。

---

### 问题 3：色溢（Color Bleeding）不工作，彩色墙壁的光弹射出来是白色的！

**为什么**：烘焙大师（Progressive Lightmapper）和我们的 Shader 是两个**完全独立不通气的大脑**。大师不认识我们自定义的 `_BaseColor`，默认当白墙算，弹射出去的光自然是白色的。  
**怎么解决**：写 **Meta Pass**（`LightMode = "Meta"`），专门开一扇只有烘焙大师才能敲的门，让大师通过这个通道来问"你的材质是什么颜色"，我们通过 BRDF 计算出漫反射颜色告诉它。  
**效果**：重新烘焙后，红色墙壁旁边的白色地板出现了粉红色晕染——真正的色溢！🌹

---

### 问题 4：代码重复——每个 Pass 都复制粘贴一堆材质变量！

**为什么**：`LitPass`、[ShadowCasterPass](cci:1://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Editor/CustomShaderGUI.cs:152:4-167:5) 各自写了一份 `TEXTURE2D(_BaseMap)` 和 `UNITY_INSTANCING_BUFFER`，维护起来一改就要改三遍。  
**怎么解决**：重构！  
- 新建 [LitInput.hlsl](cci:7://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Shaders/LitInput.hlsl:0:0-0:0) / [UnlitInput.hlsl](cci:7://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Shaders/UnlitInput.hlsl:0:0-0:0) 作为**材质数据银行**
- 在 [Lit.shader](cci:7://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Shaders/Lit.shader:0:0-0:0) / [Unlit.shader](cci:7://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Shaders/Unlit.shader:0:0-0:0) 加 **`HLSLINCLUDE` 广播大喇叭**  
- 所有 Pass 统一调用 `GetBase()`、`GetEmission()` 等接口函数  
**效果**：以后加新属性只改银行文件，代码架构清晰整洁。

---

### 问题 5：自发光物体无法把光照贡献给周围的静态物体！

**为什么**：发光的霓虹灯屏幕在现实中会把橙色光散射到周围，但我们只是在 Shader 里把颜色加在自己身上（`color += GetEmission()`），没有告诉烘焙大师。  
**怎么解决**：在 Meta Pass 里加 Y 标志位逻辑；在 [CustomShaderGUI](cci:2://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Editor/CustomShaderGUI.cs:7:0-238:1) 里加 [BakedEmission()](cci:1://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Editor/CustomShaderGUI.cs:169:4-189:5) 显示 `Global Illumination` 开关。  
**效果**：把霓虹灯物体设为 Static + GI = Baked，重新烘焙，周围白色地板上出现橙色晕。

---

### 问题 6：半透明物体的烘焙阴影是实心黑色矩形！

**为什么**：烘焙大师只认识写死的三个变量名 `_MainTex`、`_Color`、`_Cutoff`，不认识我们的 `_BaseMap` 和 `_BaseColor`，默认当实心白色方块处理。  
**怎么解决**：在 Shader 的 Properties 里加两个 `[HideInInspector]` 马甲变量，在 [CustomShaderGUI](cci:2://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Editor/CustomShaderGUI.cs:7:0-238:1) 里加 [CopyLightMappingProperties()](cci:1://file:///e:/unityProject/SRP_started/Assets/Custom%20RP/Editor/CustomShaderGUI.cs:191:4-212:5) 自动同步。  
**效果**：镂空贴图的阴影变成圆形而非矩形——烘焙器正确认出了贴图形状！

---

### 番外：MeshBall 实例接收不到 GI！

**为什么**：`DrawMeshInstanced` 是手动 Draw Call，没有 MeshRenderer 组件，Unity 的自动探针系统完全绕过了。  
**怎么解决**：手动调用 `LightProbes.CalculateInterpolatedLightAndOcclusionProbes()` 批量查探针，结果填进 `MaterialPropertyBlock`；或用 LPPV 体积代理方案。  
**效果**：1023 个小球各自按自己的位置正确显示 GI 颜色。

## 🔬 深度补充：光照探针工作原理全解

### 探针到底存的是什么？

探针在烘焙时，相当于在空间中的某个点"看向四面八方"，记录下从各个方向射来的光是什么颜色、多亮。

但全景球形数据太大，Unity 用**球谐（Spherical Harmonics，SH）**把这份数据压缩成 **27 个系数**存储，就像 MP3 压缩音频一样，损失一点精度换来极小的存储量。

---

### Q13：探针怎么"包围"物体？四面体是怎么构成的？

**四面体由 4 个探针构成，不包含物体本身！**

```
❌ 错误理解：物体 + 3个探针 = 四面体
✅ 正确理解：探针A + 探针B + 探针C + 探针D = 四面体
             物体只是"查询者"，找到自己在哪个四面体格子里
```

Unity 把场景里所有探针通过 **Delaunay 四面体剖分** 连成一张网格，把整个 3D 空间切割成紧密排列的四面体，就像把空间用积木块填满，只是每块积木都是四面体形状。

---

### Q14：插值用的是什么距离？

**不是欧氏距离！用的是体积比（重心坐标）！**

```
四面体 ABCD，物体 P 在里面：

wA = 子四面体 PBCD 的体积 / 总体积 ABCD
wB = 子四面体 APCD 的体积 / 总体积 ABCD
wC = 子四面体 ABPD 的体积 / 总体积 ABCD
wD = 子四面体 ABCP 的体积 / 总体积 ABCD

P 的光照 = A的光照×wA + B的光照×wB + C的光照×wC + D的光照×wD
```

P 越靠近哪个顶点，那个顶点对应的"小四面体"体积越大，权重越高。**这是纯粹的 3D 空间几何计算，不是简单的距离。**

---

### Q15：为什么跑到 10000m 外还受影响？没有衰减吗？

**是的，完全没有衰减因子。** 原因是探针存的是"空间环境氛围"，不是"点光源"。

```
点光源：从一个点向外辐射 → 距离越远越暗 → 需要衰减
环境光：来自四面八方的墙壁、天空、地板反射 → 整个房间氛围一样 → 不需要衰减
```

当物体在探针包围盒**外部**时（比如飞到空中比所有探针都高）：
```
→ Unity 找最近的四面体"面"（三角形），退化成 3 个探针插值（2D 重心坐标）
→ 无论跑多远，永远用最近边界的数据外推
→ 所以 10000m 还有颜色影响！
```

**你亲眼验证的截图**：球在所有探针上方 → 只连接了下方 3 个探针形成三角形 → 这就是外推模式！

当你在上方加一个探针，球就会被 4 个探针包围成四面体内部，变成正常的体积插值模式。

---

### Q16：为什么有些场景探针密密麻麻？

球谐是**低频近似**，四面体内部的插值是**线性的**：

- 四面体越大（探针稀）→ 插值过渡粗糙，颜色可能突变
- 四面体越小（探针密）→ 插值过渡平滑，颜色梯度自然

**经验法则**：
- 光照变化剧烈的地方（阴影边界、颜色交界）→ 密集放探针
- 光照均匀的开阔地 → 稀疏就够

**最佳实践：上下各放一层探针！**

```
        ● ← 上层探针（高于动态物体）
       /|\
      / | \
     /  物体  \
    /    |    \
   ●─────┼─────●  ← 下层探针
          |
          ●
```

这样物体始终在四面体"内部"，避免退化到外推模式。
